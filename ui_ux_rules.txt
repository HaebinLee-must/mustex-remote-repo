# CEX UI / UX Rules & Design System (Skills용)

## 1. 문서 역할
본 문서는 CEX Core PRD를 전제로, 서비스 전반에 적용되는 UI·UX 판단 규칙과 디자인 시스템 성격의 스타일 가이드를 정의한다.

본 문서는 “컴포넌트를 어떻게 그릴지”가 아니라  
**“어떤 상황에서 어떤 UI 판단을 해야 하는지”를 고정하는 문서**다.

---

## 2. 디자인 기준 출처 및 우선순위

### 2.1 기준 우선순위
1. **기제작 앱 UI (KYC 플로우 캡처 + 첨부 tsx 컴포넌트)**
2. 바이낸스 UI (정보 밀도·거래 화면 구조 참고용)
3. 일반적인 웹 UI 패턴

### 2.2 차용 원칙
- 색상 톤, 키 컬러, 분위기 → **기제작 앱 기준**
- 정보 배치 밀도, 테이블 구성, 패널 구조 → **바이낸스 참고**
- 브랜드 컬러, 포인트 컬러는 바이낸스를 따라하지 않는다.

---

## 3. UI·UX 핵심 철학 (CEX 공통)

- 정보는 숨기지 않는다.
- 행위는 제한할 수 있다.
- UI 노출과 실행 가능 여부는 분리한다.
- “지금 무엇을 해야 하는지”를 항상 알 수 있어야 한다.
- 금융 UI는 감성보다 **신뢰·안정·명확성**을 우선한다.

---

## 4. 컬러 시스템 규칙 (의미 중심)

### 4.1 컬러 사용 원칙
- 컬러는 **의미를 가진다**
- 장식 목적의 컬러 사용을 금지한다.
- 모든 색상은 토큰 기반으로 사용한다.

### 4.2 키 컬러 기준
- **Primary 컬러는 기제작 앱 및 tsx 기준을 절대 기준으로 한다.**
- 바이낸스의 옐로우 계열 컬러는 사용하지 않는다.

### 4.3 컬러 의미 체계
- Primary: 주요 행동(Continue, Confirm), 현재 단계 강조
- Surface 계열: 카드, 입력 영역, 패널
- Accent: 보조 강조, 추천, 가이드 표시
- Success / Warning / Error: 상태·결과 전달 전용

---

## 5. 타이포그래피 규칙

### 5.1 기본 원칙
- 가독성이 브랜드보다 우선한다.
- 숫자 정보는 텍스트보다 더 명확해야 한다.
- 대문자 남용 금지.

### 5.2 계층 구조
- Page Title: 명확하고 짧게 (예: “Let’s Get You Verified”)
- Section Label: 작은 크기, 낮은 대비
- Primary Content: 가장 높은 대비
- Helper / Description: 명확하되 시각적 우선순위는 낮게

---

## 6. 레이아웃 및 밀도 규칙

### 6.1 기본 레이아웃
- 어두운 배경 + Surface 카드 구조를 기본으로 한다.
- 한 화면에는 하나의 주요 목적만 둔다.
- KYC, 설정, 입력 플로우는 **중앙 정렬 단일 컬럼**을 기본으로 한다.

### 6.2 정보 밀도
- 거래 화면은 고밀도 허용
- 온보딩 / KYC / 설정 화면은 저밀도 유지
- 바이낸스의 밀도는 **Trade 화면에서만 참고**한다.

---

## 7. 입력 필드(Input / Select) 규칙

### 7.1 입력 필드 스타일
- Input은 항상 Surface 위에 존재한다.
- Placeholder는 정보가 아니라 힌트다.
- Focus 상태는 명확히 구분하되 과하지 않게 한다.

### 7.2 Select / Dropdown
- 기본 상태: 닫힘
- 선택 후: 값이 명확히 고정되어 보이도록
- 국가, 문서 유형 등은 **정책에 따라 옵션이 달라질 수 있음을 암시**한다.

---

## 8. 선택형 리스트 (Document Type 등)

### 8.1 카드형 선택 패턴
- 문서 유형 선택은 **카드형 리스트**를 기본으로 한다.
- 선택 가능 여부는 시각적으로 즉시 인지 가능해야 한다.

### 8.2 Recommended 표시
- Recommended는 Accent 계열 컬러로만 표현
- 강제 선택처럼 보이게 하지 않는다.

---

## 9. 버튼 규칙 (Button.tsx 기준)

### 9.1 버튼 역할 구분
- Primary Button: 한 화면에 하나
- Secondary Button: 보조 행동
- Disabled Button: 숨기지 않는다

### 9.2 Disabled 상태 UX
- 클릭 불가 사유가 존재해야 한다.
- 필요 시 클릭 또는 Hover로 사유 안내 가능

---

## 10. Stepper / Progress 규칙

### 10.1 Stepper 역할
- Stepper는 “진행 상황 인지용”이다.
- 상태(State)와 1:1 매핑되지 않아도 된다.

### 10.2 표현 규칙
- 현재 단계는 Primary로 강조
- 완료 단계는 Success 톤
- 미도달 단계는 낮은 대비

---

## 11. 상태 / 제한 / 안내 UX

### 11.1 제한 상태 표현
- 단순 비활성 처리 금지
- 반드시 “왜 불가한지”를 설명한다.

### 11.2 제한 사유 분류
- 상태(State)
- 정책(Policy)
- 리스크(Risk)

UI 문구에서 이 구분이 드러나야 한다.

---

## 12. Trade UI 참고 규칙 (Binance 차용 범위)

### 12.1 차용 가능한 요소
- 패널 분할 구조
- 테이블 컬럼 정렬 방식
- 차트 + 주문 영역 분리 패턴

### 12.2 차용 금지 요소
- 바이낸스 컬러 시스템
- 과도한 강조색 사용
- 지나치게 작은 폰트 사이즈

---

## 13. 아이콘 규칙

- 아이콘은 보조 수단이다.
- 텍스트 없이 의미 전달 금지
- lucide 계열과 유사한 단순 스트로크 스타일 유지

---

## 14. AI 생성 시 금지 UX 패턴

AI는 다음을 생성해서는 안 된다.
- 브랜드 컬러를 무시한 임의 색상 사용
- 금융 서비스에 부적절한 장식적 UI
- 제한 사유 없는 비활성 버튼
- 바이낸스 UI를 그대로 모사한 화면

---

## 15. 요약 기준 문장
- 이 서비스의 UI는 “신뢰 가능한 금융 도구”처럼 보여야 한다.
- 스타일은 기제작 앱을 따른다.
- 구조와 밀도는 CEX 표준을 따른다.
- 모든 UI 판단은 Account 중심 구조를 훼손하지 않아야 한다.

본 문서는 CEX UI·UX Rules 및 디자인 시스템의 기준 문서다.

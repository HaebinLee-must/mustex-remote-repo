# CEX UI/UX Rules

## 1. 기본 철학

### 1.1 정보 투명성
- 정보는 숨기지 않는다.
- 행위는 제한할 수 있다.
- 사용자는 자신의 상태를 항상 인지할 수 있어야 한다.

---

### 1.2 제한 상태 표현 원칙
- 단순 비활성화는 허용하지 않는다.
- 제한 사유를 명확히 안내한다.
- 제한 원인이 상태(State), 정책(Policy), 리스크(Risk) 중 무엇인지 구분한다.

---

## 2. 제한 상태 UX 패턴

### 2.1 State 기반 제한
**표현 방식:**
- "이 기능을 사용하려면 KYC 인증이 필요합니다"
- 다음 단계로 이동할 수 있는 명확한 CTA 제공

---

### 2.2 Policy 기반 제한
**표현 방식:**
- "현재 국가에서는 이 기능을 사용할 수 없습니다"
- 정책 변경 가능성이 있는 경우 알림 신청 옵션 제공

---

### 2.3 Risk 기반 제한
**표현 방식:**
- "보안 검토가 진행 중입니다. 고객센터에 문의해주세요"
- 자동 해제 불가능함을 명시

---

## 3. 환경 불안정성 대응 UX

### 3.1 네트워크 실패 처리
- 로딩 상태를 명확히 표시
- 타임아웃 발생 시 재시도 옵션 제공
- 실패 원인을 사용자가 이해할 수 있는 언어로 표현

---

### 3.2 재시도 UX
- 재시도 버튼을 명확히 제공
- 재시도 횟수 제한이 있는 경우 명시
- 자동 재시도 시 사용자에게 진행 상황 표시

---

## 4. Pilot 단계 UX 원칙

### 4.1 기능 제한 안내
- "현재 Pilot 운영 중으로 일부 기능이 제한됩니다"
- 제한 해제 예정 시기를 명시 (가능한 경우)

---

### 4.2 피드백 수집
- Pilot 단계에서는 사용자 피드백 수집 UI를 적극 노출
- 피드백이 서비스 개선에 반영됨을 명시

---

## 5. 금지 패턴

다음 UX 패턴은 사용하지 않는다:
- 이유 없는 비활성화
- "준비 중입니다" 같은 모호한 메시지
- 에러 코드만 표시하고 설명 없는 실패 메시지
- 사용자 액션 없이 자동으로 사라지는 중요 알림
